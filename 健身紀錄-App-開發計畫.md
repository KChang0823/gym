# 健身紀錄 App 開發計畫

## Turn 1

### 👤 User

我現在想做一個簡易紀錄我去健身房次數與時長的app
流程圖：
iphone捷徑 --> 選擇入場/出場 -->
1. 跳轉連結： https://members.starfitness.com.tw/#/index/main
2. 紀錄當下日期與時間
用簡單的one-page html寫一個前端顯示每月該去的時數是否達標（用github pages deploy)
因為每月988元，所以假設1分鐘1.2元(可變動)的狀況下我必須去多久我才有值回票價
另外，你覺得後端要用什麼？（google sheets?supabase?)
幫我寫一份完整且詳細的計劃

### 🤖 Assistant

這個自己動手做（DIY）的追蹤工具非常實用。要把 iOS 捷徑、資料庫和前端網頁串得乾淨俐落，我們需要一個低維護成本且無縫接軌的架構。
針對你的問題，**我強烈建議後端使用 Google Sheets 搭配 Google Apps Script (GAS)**，而不是 Supabase。
原因很簡單：這是一個輕量級的個人時序紀錄（Time-series logging）應用。Google Sheets 完全免費、不用處理麻煩的 CORS 或 API 金鑰驗證，而且 GAS 原生就能部署成 Webhook (Web App)，這讓 iOS 捷徑透過 `POST` 請求傳遞資料變得極度無腦。既然你對使用 GAS 開發網頁應用的節奏並不陌生，這會是阻力最小、最快上線的方案。
以下是為你規劃的完整開發步驟：

### 第一階段：後端 (Google Sheets + GAS)
這是整個系統的大腦，負責接收捷徑傳來的打卡資料，並提供前端所需的加總數據。

1. **建立 Google 試算表**
  - 建立三個工作表（Tabs）：`RawData`（存原始打卡紀錄）、`DailySummary`（每日加總）、`MonthlySummary`（每月加總）。
  - `RawData` 欄位設計：`ID` (UUID 或 Timestamp)、`日期`、`動作` (入場/出場)、`時間`。
2. **撰寫 Google Apps Script**
  - 寫一個 `doPost(e)` 函數來接收 iOS 捷徑傳來的 Webhook。
  - **邏輯判斷**：
    - 如果收到的動作是「入場」：在 `RawData` 寫入一筆新紀錄。
    - 如果收到的動作是「出場」：在 `RawData` 寫入出場紀錄。
  - 寫一個 `doGet(e)` 函數當作 API 給前端呼叫。這個 API 會計算並回傳「本月累積健身分鐘數」。
3. **部署**
  - 將 GAS 部署為「網頁應用程式」，存取權限設定為「所有人」，取得專屬的 Web App URL。

### 第二階段：iOS 捷徑 (資料收集與流程控制)
這是你的 UI，重點是快、不用動腦。

1. **建立選單**：在捷徑加入「從選單選擇」，提供兩個選項：「💪 入場」與「👋 出場」。
2. **處理入場邏輯**：
  - **觸發 Webhook**：使用「取得網址內容」動作，將網址填入 GAS 的 Web App URL，方法改為 `POST`，並在要求主體（JSON）中帶入 `{"action": "entry", "timestamp": "目前日期與時間"}`。
  - **跳轉網頁**：使用「打開 URL」動作，帶入 `https://members.starfitness.com.tw/#/index/main` 讓你刷條碼進場。
3. **處理出場邏輯**：
  - **觸發 Webhook**：一樣使用「取得網址內容」打 `POST` 到 GAS，主體帶入 `{"action": "exit", "timestamp": "目前日期與時間"}`。

### 第三階段：前端 (One-Page HTML + GitHub Pages)
用最簡單的 HTML/CSS/Vanilla JS 寫一個乾淨的儀表板。

1. **核心數值設定**：
  - 月費：$988
  - 每分鐘價值：$1.2（寫成變數 `const valuePerMinute = 1.2;` 方便隨時調整）
  - **目標分鐘數**：$988 \div 1.2 \approx 823.3$ 分鐘（約 13.7 小時）。
2. **版面設計**：
  - **大字體顯示**：本月已累積時間（X 小時 Y 分鐘）。
  - **進度條 (Progress Bar)**：計算 `(已累積分鐘數 / 824) * 100` 的百分比。
  - **回本狀態**：顯示目前已經「賺回」多少錢（累積分鐘數 * 1.2），以及還差多少達標。
3. **資料串接**：
  - 使用 `fetch()` 呼叫你的 GAS `doGet` URL，取得本月累積分鐘數，並即時渲染到畫面上。

### 第四階段：部署

- 將寫好的 `index.html`（及相關的 CSS/JS）推送到 GitHub repository。
- 在設定中開啟 GitHub Pages，指向 `main` 分支，你的儀表板就正式上線了。

### 💡 來自 Sparring Partner 的誠實建議
從工程角度來看，這個計畫非常完美。但從「健身」的角度來看，我想稍微挑戰一下你的**核心指標（Metric）**。
你將「待在健身房的時間」換算成「金錢價值（1分鐘 1.2元）」，這在經濟學上叫沉沒成本的回收焦慮。**健身房的價值在於訓練品質與強度（例如：你的深蹲、硬舉 PR 有沒有突破），而不是待在裡面的時長。** 如果為了讓網頁上的進度條達標（湊滿那 13.7 小時），而在組間休息時狂滑手機、延長滯留時間，反而會讓訓練品質下降，本末倒置。
**你可以怎麼優化？**
這個 App 依然很棒，但你可以把它當作**「一致性（Consistency）」**的指標，而不是「回本」的計算機。確保自己每週穩定去 3 到 4 次（完美契合你的上下半身分化訓練），只要有踏進去並完成課表，這 988 元就已經發揮了最大的價值。
